# Idempotent Jobs

冪等なジョブの設計パターン。ジョブが初回で成功しても、失敗後のリトライで成功しても、システムの最終状態が同じになるように設計する。

## 概要

失敗したジョブのリトライ機能を最大限に活用するには、ジョブを冪等（idempotent）に設計する必要がある。つまり、初回成功でもリトライ成功でもシステムの最終状態に違いが出ないようにすること。

冪等性を実現するための原則:

- ジョブはできるだけ **アトミック** かつ **シンプル** に保つ
- 複数の異なるアクション（データベース更新、API 呼び出しなど）を1つのジョブで同時に実行しない
- プロセスフローの追跡が困難になり、例外発生時の部分的なロールバックが難しくなる

シンプルなジョブはデバッグやボトルネックの特定も容易になる。

複雑なジョブが必要な場合は、[フローパターン](./flows.md)で分割することを検討する。

## 注意点

- リトライ可能なジョブは必ず冪等に設計すること
- 1つのジョブに複数の副作用を含めない
- 外部 API 呼び出しには冪等キーの利用を検討する
- データベース操作には upsert やトランザクションを活用する
