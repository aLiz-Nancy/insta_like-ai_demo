# Webhook ベストプラクティス

## レスポンス速度

- **タイムアウト**: サーバーは Webhook 配信を受信してから **10 秒以内** に 2XX レスポンスを返す必要がある
- タイムアウトすると GitHub は配信失敗として記録する
- ペイロードの処理は非同期で行い、受信確認のみ即座に返すこと

## 非同期処理

Webhook ペイロードはキューに入れてバックグラウンドで処理することを推奨。

### 推奨ツール

| 言語/環境 | ツール |
|----------|--------|
| 汎用 | Hookdeck |
| Ruby | Resque |
| Python | RQ (Redis Queue) |
| Java | RabbitMQ |
| Node.js | Bull, BullMQ |

### 処理フロー（推奨）

```
1. GitHub からの Webhook 受信
2. 署名を検証
3. 2XX レスポンスを即座に返す
4. ペイロードをキューに追加
5. ワーカーがキューからペイロードを取得し処理
```

## イベントサブスクリプション

- 必要なイベントのみをサブスクライブしてサーバー負荷を軽減する
- 不要なイベントを受信すると、処理コストとネットワーク帯域が無駄になる

## イベント検証

ペイロードを処理する前に、以下を確認すること:

- `X-GitHub-Event` ヘッダーでイベント種別を確認
- ペイロード内の `action` キーでアクション種別を確認
- 想定外のイベント/アクションは無視する

## 再配信と冪等性（Idempotency）

- サーバー復旧後、GitHub UI から未配信の Webhook を再配信できる
- 再配信されたイベントは元の `X-GitHub-Delivery` ヘッダー値を保持する
- **冪等性の確保**: `X-GitHub-Delivery` を使用して重複配信を検出し、同じイベントを二重処理しないようにする

```
処理済み配信 ID を記録:
1. X-GitHub-Delivery を取得
2. 既に処理済みかチェック
3. 未処理なら処理を実行し、ID を記録
4. 処理済みならスキップ
```

## セキュリティ

### Secret の管理

- 高エントロピーなランダム文字列を使用する
- Secret は環境変数や Secret Manager に安全に保管する
- ペイロード URL に機密情報を含めない

### HTTPS/SSL

- HTTPS 接続を使用し、SSL 検証を有効にする
- 自己署名証明書は避ける

### IP アドレス制限

- `GET /meta` エンドポイントで GitHub の IP アドレス範囲を取得できる
- ファイアウォールやアクセス制御リストで GitHub からのリクエストのみを許可する
- IP アドレスは変更される可能性があるため、定期的に更新すること

### リプレイ攻撃の防止

- `X-GitHub-Delivery` ヘッダーを利用して各配信がイベントごとに一意であることを確認する
- タイムスタンプの検証を追加し、古い配信を拒否することも有効

## エラーハンドリング

- 配信失敗時、GitHub は再試行を行う
- 失敗した配信は GitHub UI の **Recent Deliveries** で確認できる
- 配信の再送は手動で行うことも可能

## 監視

- Webhook の配信状態を定期的に監視する
- 失敗率が高い場合はサーバーやネットワークの問題を調査する
- GitHub の Webhook ログ（Recent Deliveries）を活用する
