# セキュリティ概要

## 責任共有モデル

Supabase とユーザーの間でセキュリティ責任が分担される。

### Supabase の責任

- インフラストラクチャのセキュリティ（物理、ネットワーク）
- プラットフォームのセキュリティ（パッチ適用、脆弱性管理）
- データの暗号化（保存時・転送時）
- サービスの可用性
- SOC2 Type II 準拠
- 定期的なペネトレーションテスト

### ユーザーの責任

- RLS ポリシーの設計と実装
- API キーの管理
- アプリケーションレベルのセキュリティ
- ユーザーデータの適切な取り扱い
- アクセス制御の設定
- バックアップとリカバリの計画

## プラットフォームセキュリティ

### SOC2 Type II

- Supabase は SOC2 Type II 認証を取得
- 独立した監査法人による継続的な監査
- セキュリティ、可用性、機密性の統制を証明

### インフラストラクチャ

- AWS 上でホスティング
- データは保存時に AES-256 で暗号化
- 転送時は TLS 1.2 以上で暗号化
- VPC 内で隔離されたネットワーク
- DDoS 保護

### プラットフォームの運用セキュリティ

- 自動パッチ適用
- 脆弱性スキャン
- インシデント対応プロセス
- セキュリティログの監視

## プロダクトセキュリティ

### RLS（Row Level Security）

Supabase のセキュリティモデルの中核。PostgreSQL のネイティブ RLS を使用してテーブルレベルのアクセス制御を実現する。

```sql
-- RLS の有効化
ALTER TABLE public.posts ENABLE ROW LEVEL SECURITY;

-- ポリシーの設定
CREATE POLICY "Authenticated users can read"
  ON public.posts FOR SELECT
  TO authenticated
  USING (true);
```

**重要**: `public` スキーマのすべてのテーブルで RLS を有効にすること。RLS が無効なテーブルは API 経由で誰でもアクセス可能。

### API キー

| キー | 用途 | 公開可否 |
|------|------|---------|
| anon key | クライアント側 API アクセス | 公開可（RLS で保護） |
| service_role key | サーバー側管理操作 | 絶対に非公開 |

#### anon key

- RLS ポリシーに従ったアクセスのみ許可
- クライアントアプリケーションに埋め込み可能
- JWT の `role` クレームが `anon`

#### service_role key

- RLS をバイパスして全データにアクセス可能
- サーバーサイドのみで使用
- 環境変数で管理し、ソースコードに含めない
- JWT の `role` クレームが `service_role`

### データベースパスワード

- プロジェクト作成時に設定
- 直接接続（ポート 5432）で使用
- 強力なパスワードを使用すること

### JWT シークレット

- API キーの署名に使用
- デフォルト値から必ず変更すること（自己ホスティングの場合）

## セキュリティチェックリスト

### 必須

- [ ] すべてのテーブルで RLS を有効化
- [ ] 適切な RLS ポリシーを設定
- [ ] service_role key をクライアントに公開していない
- [ ] データベースパスワードが強力
- [ ] 環境変数でシークレットを管理

### 推奨

- [ ] ネットワーク制限を設定
- [ ] ダッシュボードに MFA を設定
- [ ] カスタムの JWT シークレットを使用
- [ ] SSL 接続を強制
- [ ] 定期的にセキュリティ監査を実施
- [ ] 不要な API エンドポイントを無効化
- [ ] CORS を適切に設定

## セキュリティ関連ツール

### supabase db lint

```bash
# セキュリティ問題を検出
supabase db lint
```

RLS が無効なテーブルやポリシー未設定のテーブルを検出できる。

### ダッシュボードの Security Advisor

ダッシュボードの **Database → Security Advisor** でセキュリティ推奨事項を確認できる。
